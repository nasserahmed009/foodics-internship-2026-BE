Foolix Mock API

The backend for the Foolix ordering app — a foul & ta3meya restaurant. This reference lists every endpoint, its request and response shapes, and the types of every model.

Overview

  • Base URL  http://localhost:4000
  • Format  application/json
  • Currency  EGP
  • Tax  14% VAT

Send and receive JSON. Set Content-Type: application/json on requests with a body. Every endpoint except POST /login requires a bearer token — see below. List endpoints support filtering, search, sorting and pagination via query parameters (documented per endpoint).

This is a mock server (json-server). It persists to a file, adds artificial latency, and can inject random failures — it's built to make the frontend handle real-world states (loading, empty, error), not to be a production backend.

Authentication

Call POST /login to get a token, then send it on every other request:

Authorization: Bearer <token>

Requests without a valid bearer token get 401 Unauthorized. The token is a fake JWT — the mock doesn't verify signatures, it only checks the header is present. Two seed users exist (any non-empty password works):

EmailRoleLands on
cashier@foolix.testcashierCashier / POS area
admin@foolix.testadminOperations Dashboard

The frontend reads user.role from the login response and redirects accordingly.

Data types

The app is written in JavaScript; the shapes below are written as TypeScript-style interfaces purely to document each model precisely.

Enums

type Role        = "cashier" | "admin"
type OrderType   = "dine_in" | "takeaway" | "delivery"
type OrderStatus = "pending" | "preparing" | "ready" | "completed" | "cancelled"
pending preparing ready completed cancelled
Order status normally advances pending → preparing → ready → completed (or → cancelled). The mock accepts any status via PATCH; the frontend enforces the allowed flow.

User

interface User {
  id: number
  name: string
  email: string
  role: Role
}

Category

interface Category {
  id: number
  name: string        // e.g. "Foul"
  nameAr: string      // Arabic name, e.g. "فول"
  sortOrder: number
}

Modifier & Product

interface Modifier {
  id: number
  name: string        // "Extra tahini"
  price: number       // added to the item price; can be 0
}

interface Product {
  id: number
  name: string
  nameAr: string
  description: string
  price: number       // base price in EGP
  categoryId: number  // -> Category.id
  imageUrl: string
  available: boolean
  modifiers: Modifier[]
}

Order & OrderItem

interface OrderItem {
  productId: number
  name: string
  qty: number
  unitPrice: number                          // base product price
  modifiers: { name: string; price: number }[]
  lineTotal: number                          // (unitPrice + Σ modifier.price) × qty  — computed by the server
}

interface Customer {
  name: string
  phone: string
  address?: string
}

interface Order {
  id: number
  number: number         // 4-digit human-facing number, e.g. 1042
  status: OrderStatus
  type: OrderType
  table: number | null   // set for dine_in
  customer: Customer | null   // set for delivery
  items: OrderItem[]
  subtotal: number       // Σ lineTotal        — computed
  tax: number            // subtotal × 14%     — computed
  total: number          // subtotal + tax     — computed
  notes: string
  createdAt: string      // ISO timestamp
}

Login response

interface LoginResponse {
  token: string
  user: User
}

Endpoints · Auth

POST /login Auth: none

Exchange credentials for a token and the signed-in user (with role). Any non-empty password is accepted for a known seed email.

  "email": "cashier@foolix.test",
  "password": "demo"
}

  "token": "fake-jwt.1.1783175087704",
  "user": {
    "id": 1,
    "name": "Sara Adel",
    "email": "cashier@foolix.test",
    "role": "cashier"
  }

Endpoints · Orders

POST /orders Auth: bearer

Create an order. Send items only — the server computes each lineTotal, the subtotal, tax and total, assigns the order number, sets status: "pending" and createdAt. Any totals you send are ignored, so always read them back from the response.

Request body 
{
  "type": "dine_in",
  "table": 9,
  "notes": "Extra napkins please",
  "items": [
    {
      "productId": 1,
      "name": "Foul Eskandarani",
      "qty": 2,
      "unitPrice": 25,
      "modifiers": [
        { "name": "Extra tahini", "price": 5 },
        { "name": "Add egg", "price": 8 }
      ]
    },
    { "productId": 12, "name": "Tea", "qty": 1, "unitPrice": 8, "modifiers": [] }
  ]
}
For delivery orders send a customer object instead of table: 
"type": "delivery",
"customer": { "name": "Nadia M.", "phone": "0100-123-4567", "address": "12 El Nasr St, Cairo" }

PATCH /orders/:id Auth: bearer

Advance an order's status (or cancel it). This is what the dashboard's status buttons call — a good place for an optimistic UI update.

Request body 
{ "status": "preparing" }
200 · Response 
{
  "id": 2, "number": 1043, "status": "preparing", "type": "takeaway",
  "items": [ /* … */ ],
  "subtotal": 50, "tax": 7, "total": 57,
  "createdAt": "2026-07-04T07:32:00.000Z"
}

Errors

All errors return JSON with a message. Handle these states in the UI.

StatusWhenBody
400Invalid order payload (no items, bad qty, invalid type){ "message": "…" }
401Missing/invalid bearer token, or bad login{ "message": "Unauthorized — missing or invalid token" }
404Resource id doesn't exist{}
500Injected failure when FAIL_RATE > 0 (writes only){ "message": "Something went wrong. Please try again." }
Set FAIL_RATE=0.1 when starting the server to make ~10% of writes fail, so you can build and test retry / error UX.

Configuration

Behaviour is controlled by environment variables when starting the server.

VariableDefaultPurpose
PORT4000Port to listen on.
TAX_RATE0.14VAT applied to order subtotals.
LATENCY_MIN_MS / LATENCY_MAX_MS150 / 550Artificial latency range.
FAIL_RATE0Probability (0–1) a write returns 500.
API_URL / INTERVAL_MSlocalhost:4000 / 20000Used by the order-injector script.
Run 
npm install
npm start              # http://localhost:4000
FAIL_RATE=0.1 npm start   # with injected failures
npm run inject         # (2nd terminal) a new order every ~20s
